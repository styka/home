2. Analizy podstawowe: UX, baza danych, skalowanie

Analiza UX: Keyboard-first, vim-style, dark theme

Punkty silne

✓ Keyboard shortcuts (vim style): j/k (nawigacja), x/Space (toggle status), e (edit), d (delete), a/n (add). Dla power users to ogromna produktywność — zero myszy, flow state. Consistency z VS Code, Linear, GitHub.
✓ Dark theme z CSS variables: Ciemne kolory (--bg-base: #0d0d0d, --text-primary: #ffffff) zapewniają czytelność i zmęczenia oczu. Skin system (user/team/system skins) pozwala personalizację bez hardcoded kolorów.
✓ Minimalist design (Linear/GitHub aesthetic): Spacing, contrast, readable font stack (San Francisco / Segoe). Brak zbędnych animacji, fokus na struktura i funkcjonalność.

Punkty słabe

⚠ Onboarding: Brak tutorial flow — użytkownik ląduje w Dashboard AI (Home) bez wskazówek. Novice users mogą nie wiedzieć gdzie kliknąć (mobile może być szczególnie trudne). Rekomendacja: interactive tutorial (skip-able).
⚠ Mobile responsiveness: Sidebar poza ekranem (md:hidden), bottom tab bar pojawia się. Ale: niektóre moduły (Storage Pro, Workshops Pro) mają interface bliżej desktop-first. Mobile UX wymaga polishing.
⚠ Accessibility (a11y): Keyboard shortcuts świetne dla sighted users. Brak ARIA labels na bogatych widokach (drag-drop, grid). Screen readers mogą mieć kłopoty na mobilnych interaktywnych widokach (Storage scan, Kitchen plan).

Rekomendacje (do dyskusji w debatach)

  • Onboarding: First-time user tutorial (5 min, skip-able, contextual hints w każdym module).
  • Mobile: Audit każdego modułu na iPhone SE (mały ekran), zapewnić touch targets ≥48px.
  • a11y: ARIA labels, keyboard-only navigation, high contrast mode toggle.

Analiza bazy danych: Schemat, indeksy, skalowanie

Stan bazy (Prisma 5 + PostgreSQL Neon free tier)

Modele: 56+ (Auth, RBAC, Teams, Shopping, Tasks, Notes, Kitchen, Pets, Health, Habits, Flota, Portfel, Languages, Services, Storage, Workshops, QA, LLM, System).

Migracje: 121 (systematyczne podejście — każda migracja dokumentowana, reversible).

Punkty silne

✓ Schema-first z Prisma: Deklaratywny, type-safe, generuje TypeScript types. Każdy Server Action zna kształt danych — zero runtime surprises.
✓ Relationships dobrze zaprojektowane: User/Team ownership (ownerId / ownerTeamId), FK constraints, ON CASCADE DELETE gdzie sensowne. RBAC przez UserRole + Permission + RolePermission.
✓ Multi-tenancy (Teams): Wszystkie zasoby mogą być user- lub team-scoped. Query pattern: where: { OR: [{ ownerId }, { ownerTeamId }] } — skalowalne.

Punkty słabe & wyzwania

⚠ N+1 queries: Niektóre page.tsx wczytują liste (np. items) bez include / select dla related danych. Np. Shopping list z 100 items × N category lookups = N+1. Z czasem (10k items) będzie wolne.
⚠ Brak indeksów composite na hot queries: Queries takie jak findMany({ where: { ownerId, status } }) mogą być slow bez indeksu na (ownerId, status). Neon free tier ma CPU limity.
⚠ Large JSON columns: Task.recurring (JSON), Item.metadata (JSON) — bez indeksów JSONB w Free tier Neon. Jeśli Growing — slow queries (100k+ records).
⚠ Activity tracking (UserActivity): Każda akcja loguje rekord. Przy 10k users × 10 akcji/dzień = 100k rows/dzień. Po roku: 36M rows. Query findMany({ where: { createdAt: { gte: since7d } } }) będzie slow bez indeksu na createdAt.

Rekomendacje

  • Audyt N+1 queries: Przejrzeć każdy page.tsx — dodać include: { related } lub select gdzie brakuje.
  • Composite indexes: Dodać w migracjach: @@index([ownerId, status]) dla hot tables (Item, Task, Note).
  • JSONB indexing: W Neon Pro dodać @@index([jsonColumn]) dla Task.recurring.
  • Archiwizacja ActivityLog: UserActivity starsze niż 90 dni → archive table (read-only). Przejść na event streaming (Kafka/PubSub) w Fazie 4.
  • Read replicas: Neon Pro — enable read replicas dla analytics queries (nie blokuje writes).

Analiza skalowania: Free tier → Pro tier → Enterprise

Free tier (obecnie — Render + Neon free)

Zasób Limit Działanie dziś (1–50 users)
Render (Node.js) 0.5 CPU, 512 MB RAM, spin-down po 15 min ✓ OK (cold start ~10s, ale rare)
Neon PostgreSQL 3 GB storage, shared CPU, IP allowlist ✓ OK (1–50 users nie stresuje)
Bandwidth No limit ✓ OK
Concurrent requests Limited by Node.js pool (~10–20 concurrent) ⚠️ 50 users simultaneous = timeouts possible

Bottleneck identyfikowani

  • Render free tier: CPU spikes (build, image processing) → OOM kill. Solution: vertical scaling (Render Pro: 1 CPU, 4 GB RAM, $7/mo).
  • Neon free tier: Query complexity limits. Solution: Neon Pro ($15–50/mo zależy od usage).
  • LLM API calls: Groq free tier? Czy mamy rateLimit? Jeśli 50 users × 10 calls/day = 500 calls/day — powinno być OK.

Roadmap skalowania

Q2 2026 (teraz): Free tier OK dla 1–50 users (validation phase).
Q3 2026: Upgrade do Render Pro ($7/mo) + Neon Pro ($25/mo) przy ~1k users. Cost: ~$32/mo.
Q4 2026: Load testing — jeśli 10k users, consider:
  • Redis cache layer (Upstash Redis: $1–50/mo) — session, queries cache.
  • CDN (Cloudflare: free tier wystarczy) — images, static assets.
  • Database scaling: Neon Pro → dedicated instance (~$100/mo) lub managed PostgreSQL (DigitalOcean, AWS RDS).
2027 (Faza 4): Multi-region (edge compute) — clients na EU, US, ASIA.

Performance benchmarks (target)

  • Lighthouse (Desktop): Performance ≥90, Accessibility ≥95.
  • Core Web Vitals (CWV): LCP <2.5s, FID <100ms, CLS <0.1.
  • API response time: Server Action <200ms (auth + Prisma query), LLM call <3s (streaming enabled).
  • Database query time: 95th percentile <100ms (bez LLM calls).

Podsumowanie

UX jest solidna dla power users (vim shortcuts, dark theme), ale wymaga polishing dla novice users (tutorial, mobile audit, a11y).

Baza danych jest dobrze zaprojektowana schema-wise, ale wymaga optimizacji N+1 queries, indexów composite, archiwizacji activity logs.

Skalowanie z free tier do 1–50 users jest OK dzisiaj; Q3 2026 upgrade Render Pro + Neon Pro (~$32/mo) dla 1k users; Q4 2026 — Redis + CDN.
