Zalecenie #1: Audyt N+1 queries + Redis cache

Priorytet: HIGH (performance affects scaling)
Timeline: Q2 2026 (2–3 tygodnie)
Effort: 30–40 engineer-godzin
Cost: $0 (dev time) + $5/mo (Redis)
ROI: 10x — latency -50%, scalability +3x users

Problem

Page.tsx wczytują listy bez include / select dla related danych. Np. Shopping list: 100 items × kategoria lookup = N+1 query. Przy 10k items będzie slow (100ms → 1000ms per page).

Plan dla Claude Code

Krok 1: Audit (2 dni)
  • Przejrzyj wszystkie page.tsx w src/app/ pod kątem prisma.item.findMany() bez include.
  • Uruchom Prisma Studio — slow query log (najwolniejsze queries).
  • Stwórz spreadsheet: query name, current latency (ms), missing include fields.
  • Przykład: Shopping: 
    // BEFORE (N+1)
const items = await prisma.item.findMany({ where: { listId } })

// AFTER (optimized)
const items = await prisma.item.findMany({
  where: { listId },
  include: { category: true, unit: true }
})
Krok 2: Fix queries (3–4 dni)
  • Top 10 slow queries → add include / select.
  • Dodaj composite indexes w prisma/schema.prisma: 
    model Item {
  @@index([listId, status])
  @@index([ownerId, status])
}
  • Run migration: npm run db:push (dev) / nowa migracja (prod).
  • Test: Lighthouse performance, check query times (target: <100ms 95th percentile).
Krok 3: Redis cache layer (2–3 dni)
  • Setup Upstash Redis (free tier, $5/mo for production).
  • Wrap hot queries w Caching layer: 
    // src/lib/cache.ts
import { createClient } from 'redis'
const redis = createClient({ url: process.env.REDIS_URL })

export async function getCachedItems(listId: string) {
  const cached = await redis.get(`items:${listId}`)
  if (cached) return JSON.parse(cached)

  const items = await prisma.item.findMany({...})
  await redis.setex(`items:${listId}`, 3600, JSON.stringify(items)) // 1h ttl
  return items
}
  • Invalidate cache na mutacjach: redis.del(`items:${listId}`) w createItem, updateItem, deleteItem actions.
  • Test: Performance pod load (k6 load test, 100 concurrent users).
Krok 4: Monitoring (bieżący)
  • Setup query performance monitoring (Datadog / Sentry). Alert na queries > 200ms.
  • Weekly review: Top 5 slow queries, optimize.

Expected outcome

Metryка Before After
Avg query latency 150ms 30ms (with cache)
Page load time (50 items) 600ms 150ms
Max concurrent users (free tier) 50 200+
Cost $0 +$5/mo (Redis)

Zalecenie #4: Marketplace MVP (M1–M4)

Priorytet: MEDIUM (differentiator)
Timeline: 3–4 tygodnie, Q3 2026
Effort: 100–120 engineer-godzin
Cost: $0 (dev) + Stripe + PCI DSS audit (~$500)
ROI: Sticky (5–10% users), differentiates vs Notion

MVP Scope: M1–M4 (Chat, Availability, Quotes, Portfolio)

Minimalna wersja marketplace: user (client) hires freelancer (provider) dla home service (cleaning, babysitting, handyman).

Krok 1: Schema design (1 dzień)
  • Models: 
    ServiceProvider {
  id, userId, category, title, bio, hourlyRate, profileImg, reviews
  isVerified, verificationDate
}

ServiceRequest {
  id, clientId, providerId, title, description, budget, status (pending|accepted|completed)
  createdAt, completedAt
}

ServiceQuote {
  id, requestId, providerId, quote, daysToComplete, message
  status (pending|accepted|rejected)
}

ServiceMessage {
  id, requestId, senderId, message, createdAt (chat between client & provider)
}

ServiceReview {
  id, requestId, fromUserId, toUserId, rating (1–5), comment
}
Krok 2: Routes & pages (2–3 dni)
  • /services → ServicesCatalogPage (list providers, filter by category)
  • /services/[providerId] → ProviderDetailPage (profile, reviews, "request service" button)
  • /services/my-listings → ProviderDashboard (manage profile, active requests, earnings)
  • /services/my-requests → ClientRequestsPage (list my requests, chat, tracking)
  • /services/[requestId]/chat → ChatPage (real-time messages)
Krok 3: Server Actions (2 dni)
  • createServiceProvider (user becomes provider)
  • createServiceRequest (client requests service)
  • sendQuote (provider quotes)
  • acceptQuote (client accepts, triggers payment)
  • markComplete (service done)
  • leaveReview (post-completion)
  • sendMessage (in-chat message)
Krok 4: Payments (Stripe Connect, 2 dni)
  • Setup Stripe Connect (seller accounts).
  • On quote accept: create payment intent (client card → Stripe → provider bank).
  • On completion: release funds to provider (minus commission 5%).
  • Webhook: payment.success → update ServiceRequest.status = completed.
  • ⚠️ PCI DSS audit required (can delay, but get on roadmap Q4 2026).
Krok 5: Testing & beta (2 dni)
  • Create test provider & client accounts.
  • Full flow: request → quote → payment → review.
  • Launch on beta branch (invite-only, 20–50 early adopters).
Krok 6: Decision gate (after 4 weeks beta)
  • Metric: % users requesting service > 5%? → commit to M5–M10 (etap B).
  • If < 2%? → archive marketplace (focus on core modules).

Zalecenie #7: Security (2FA, GDPR, encryption, rate limiting)

Priorytet: HIGH (compliance + trust)
Timeline: 2FA + rate limiting Q2 2026, encryption Q2 2026, GDPR Q4 2026
Effort: 50 engineer-godzin total
Cost: $0 (dev) + security audit (optional, $500)
ROI: Trust + compliance (required for enterprise)

2FA implementation (Q2 2026 — 1 week)

Krok 1: TOTP 2FA
  • Use speakeasy (TOTP library).
  • On settings page: "Enable 2FA" → generate QR code → user scans with authenticator app.
  • On login: ask for 2FA code (6 digits).
  • Store backup codes (10x single-use codes, in case user loses phone).

Rate limiting (Q2 2026 — 2 days)

Krok 1: Middleware rate limiter
  • Use Upstash Redis + @upstash/ratelimit library.
  • Limits:
    • /api/llm/* → 50 requests per day per user (prevent spam)
    • /api/auth/signin → 5 attempts per 15 min per IP (brute-force guard)
    • Server Actions → 100 per minute per user (DoS guard)

Key encryption (Q2 2026 — 2 days)

Krok 1: Encrypt API keys in Config DB
  • Use TweetNaCl.js (SecretBox encryption).
  • Master key: process.env.ENCRYPTION_KEY (environment only, never in repo).
  • On setConfigValue: encrypt before saving to DB.
  • On getConfigValue: decrypt before using.

GDPR compliance (Q4 2026 — 2 weeks)

Krok 1: Data export
  • New page: /settings/data → "Download my data" button.
  • Action: exportUserData(userId) → ZIP with JSON (all user data: items, tasks, notes, etc.).
Krok 2: Delete account
  • Action: deleteUserAccount(userId) → soft-delete (flag user.deletedAt).
  • Hard-delete after 30 days (compliance period).
Krok 3: Privacy policy + Terms
  • Update /privacy, /terms with GDPR language.
